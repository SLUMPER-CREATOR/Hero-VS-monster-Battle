Мы будем создавать текстовую RPG-игру.
Уровень 1: Базовые классы и взаимодействие.
Уровень 2: Наследование, полиморфизм и инкапсуляция.
Уровень 3: Интерфейсы, коллекции и расширяемость (SOLID).
🟢 Уровень 1: Основы (Классы и Объекты)
Тема: «Герой против Монстра»
Задача:
Создайте простую консольную программу, где есть Герой и Монстр. Они могут атаковать друг друга, пока у одного не закончится здоровье.
Требования:
Создайте класс Hero и класс Monster.
У каждого должны быть поля: Name (имя), Health (здоровье), Damage (урон).
Реализуйте метод Attack(Target target), который выводит в консоль сообщение вида: "Герой атакует Монстра и наносит 10 урона".
Здоровье цели должно уменьшаться на величину урона.
В методе Main создайте экземпляры классов и организуйте простой цикл боя (по очереди наносят удары), пока здоровье одного из них не станет <= 0.
Выведите победителя в консоль.
Критерии успеха:
Код компилируется и работает.
Логика боя понятна.
Использованы свойства (Properties) для доступа к полям.
💡 Подсказка: Не забудьте проверять, не умер ли персонаж после атаки, прежде чем передавать ему ход.
🟡 Уровень 2: Наследование и Полиморфизм
Тема: «Классы персонажей и Специализация»
Задача:
Теперь игра усложняется. Герой и Монстр — это слишком обобщенно. Нам нужны разные классы персонажей с уникальными способностями.
Требования:
Создайте базовый абстрактный класс Character (вместо отдельных Hero/Monster).
Общие поля: Name, Health, Damage.
Абстрактный метод Attack().
Виртуальный метод TakeDamage(int amount).
Создайте классы-наследники: Warrior (Воин) и Mage (Маг).
Warrior: У него есть шанс критического удара (x2 урон). Переопределите метод атаки.
Mage: У него меньше здоровья, но он может использовать заклинание Heal() (лечить себя).
Реализуйте инкапсуляцию: поля Health и Damage должны быть protected или private, доступ к ним — только через методы или свойства с валидацией (здоровье не может быть больше максимума или меньше 0).
В Main создайте список персонажей (например, 1 Воин и 1 Маг против 2 Монстров) и организуйте бой.
Добавьте простую систему уровней: после победы персонаж получает Experience и может повысить Level.
Критерии успеха:
Используется ключевое слово base для вызова методов родительского класса.
Разные классы ведут себя по-разному во время боя (полиморфизм).
Нельзя установить отрицательное здоровье извне класса.
💡 Подсказка: Для случайных событий (крит, лечение) используйте класс Random.
🔴 Уровень 3: Интерфейсы, Композиция и Архитектура
Тема: «Инвентарь, Предметы и Расширяемость»
Задача:
Игра должна стать модульной. Мы хотим добавлять новые предметы и эффекты, не переписывая классы персонажей.
Требования:
Интерфейсы:
Создайте интерфейс IUsable (используемый) с методом Use(Character target).
Создайте интерфейс IDamageable (получающий урон) для объектов, которые можно атаковать (персонажи, сундуки, двери).
Предметы:
Создайте класс Item, реализующий IUsable.
Сделайте конкретные предметы: HealthPotion (лечит), Sword (увеличивает урон при экипировке), Bomb (наносит урон по области).
Инвентарь:
Добавьте персонажам свойство Inventory типа List<Item>.
Реализуйте методы AddItem, RemoveItem, ShowInventory.
Персонаж должен иметь возможность использовать предмет из инвентаря во время боя.
События (Опционально, но желательно):
Используйте делегаты/события. Например, событие OnDeath, которое срабатывает, когда здоровье падает до 0 (чтобы вывести сообщение "Игрок погиб" в отдельном классе-логгере, а не внутри Character).
Рефакторинг:
Убедитесь, что класс Character не знает о конкретных классах Warrior или Mage при расчете урона (принцип открытости/закрытости).
Критерии успеха:
Можно добавить новый предмет (например, ManaPotion), не меняя код класса Character.
Инвентарь работает корректно (предметы удаляются после использования).
Код разбит на логические файлы (Character.cs, Items.cs, Program.cs, Interfaces.cs).
💡 Подсказка: Для реализации меча создайте свойство EquippedItem в классе персонажа, которое модифицирует урон в методе Attack.
📝 Чек-лист для самопроверки после выполнения
Чистота кода: Названия переменных понятны (health вместо h)?
Инкапсуляция: Нет ли публичных полей, которые должны быть приватными?
Повторное использование: Если я захочу добавить класс Archer, придется ли мне переписывать логику боя? (Если да, то уровень 3 выполнен не полностью).
Обработка ошибок: Что будет, если попробовать использовать зелье при полном здоровье? (Должна быть проверка).
Как приступать?
Начни с Уровня 1. Как только бой заработает, создай новый проект (или ветку в git) и переноси код туда. Уровень 2. Затем повторяй для Уровня 3.
